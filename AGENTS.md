# Lampa-Plex — контекст проекта

## Проект

Репозиторий: `KZuev/Lampa-Plex` (публичный).
Основной файл: `lampa-plex.js` — плагин Plex для медиа-интерфейса Lampa (3.0+, `Lampa.Maker`/`Lampa.Component`/`Lampa.SettingsApi`).
Общение с пользователем — **на русском**. Всегда отвечай пользователю на русском языке.

Плагин добавляет пункт бокового меню «Plex» (значок — play-в-квадрате, тот же `PLEX_PLAY_ICON`, что и у кнопки «Смотреть из Plex»; список медиатек Plex → сетка карточек фильмов/сериалов), раздел настроек «Plex» (подключение к серверу, выбор медиатек, синхронизация просмотра) и запускает воспроизведение файлов с личного Plex-сервера через `Lampa.Player` (Direct Play).

Разрабатывался по аналогии с соседним плагином [`KZuev/LampaTrakt`](https://github.com/KZuev/LampaTrakt) — оттуда взяты общие паттерны (single-file плагин, `Lampa.SettingsApi`, device-flow авторизация с QR-кодом, hybrid-интеграция с родной карточкой Lampa через `Lampa.Listener.follow('full', ...)`, синхронизация прогресса через `Lampa.Timeline`).

Коммитим и пушим **напрямую в `main`**, без PR (так же, как в LampaTrakt) — если явно не попросят иначе.

**При любом изменении плагина обновлять `PLUGIN_VERSION`** (константа в начале `lampa-plex.js`, сейчас `1.0.0`) — semver: patch (`x.x.N`) для фиксов и мелких правок, minor (`x.N.0`) для новых возможностей, major (`N.0.0`) для несовместимых/масштабных изменений. Версия видна пользователю в двух местах, которые обязаны совпадать: заголовок пункта настроек «Lampa-Plex vX.X.X» и заголовок модалки README, открываемой из него, — оба берут значение из одной константы автоматически, руками их синхронизировать не нужно.

## Публикация

`.github/workflows/pages.yml` при каждом пуше в `main` публикует корень репозитория на GitHub Pages (`actions/upload-pages-artifact` + `actions/deploy-pages`), без `enablement`/повышенных прав у токена — Pages должен быть один раз включён вручную владельцем репозитория (Settings → Pages → Source: **GitHub Actions**). Ссылка для установки в Lampa: `https://kzuev.github.io/Lampa-Plex/lampa-plex.js`.

**Важно:** не добавлять в workflow разрешение `administration: write` или `enablement: true` ради автоматического включения Pages — это расширение прав CI-токена до уровня администратора репозитория уже один раз блокировалось классификатором auto-mode как самовольное повышение привилегий. Включение Pages — осознанное действие владельца репозитория, а не то, что должен делать CI.

## Архитектура плагина (`lampa-plex.js`)

Один файл, IIFE, ES5-совместимый синтаксис, без сборки. Разделы внутри файла (сверху вниз):

- **Утилиты** — `extend`, `escapeHtml`, `buildQueryString`.
- **Хранилище** (`Lampa.Storage`): `plex_server_url`, `plex_token`, `plex_client_id` (генерируется автоматически, используется как `X-Plex-Client-Identifier`), `plex_sections_selected` (JSON-массив ключей выбранных медиатек), `plex_sync_enabled` (trigger), `plex_tmdb_index` + `plex_tmdb_index_updated_at` (персистентный индекс `tmdbId → ratingKey` всей медиатеки, см. ниже).
- **HTTP-слой**: `plexRequest`/`plexUrl` — запросы к Plex Media Server, все параметры авторизации (`X-Plex-Token` и т.д.) передаются **в query-строке**, не заголовками (чтобы не упираться в CORS-preflight на произвольных локальных адресах серверов). `plexTvGet`/`plexTvUrl` — запросы к `plex.tv` (PIN-логин, список серверов).
- **`Api`** — объект с методами `identity`, `sections`, `list` (постранично + поиск через `/library/sections/{key}/search`), `metadata`, `children` (сезоны/серии).
- **Вход через Plex.tv**: `startPlexTvLogin` → `POST /api/v2/pins` → модалка с QR (как device-auth в LampaTrakt) → поллинг `GET /api/v2/pins/{id}` → `discoverServers` (`GET /api/v2/resources`) → выбор сервера при нескольких → `applyServer` сохраняет токен+URL.
- **Настройки** (`Lampa.SettingsApi`, component `'plex'`): адрес сервера/токен вручную (`Lampa.Input.edit`, паттерн 1:1 как `trakt_client_id`/`trakt_client_secret` в LampaTrakt), кнопка входа через Plex.tv, проверка соединения, выбор медиатек (`Lampa.Select.show` с чекбоксами через повторный рендер; если `getSections()` ещё пуст — при открытии пикера `selected` предзаполняется всеми ключами из `Api.sections()`, чтобы UI визуально отражал реальное поведение по умолчанию «показываются все медиатеки», а не выглядел как «ничего не выбрано»), тумблер синхронизации, сброс настроек.
- **Меню**: `Lampa.Menu.addButton` (публичный API, а не ручной jQuery-хак в `.menu__list`, который использует LampaTrakt для совместимости со старыми версиями Lampa). Если Plex не настроен — клик по пункту меню открывает настройки плагина (`Lampa.Controller.toggle('settings'); Lampa.Settings.create('plex')`).
- **Компонент `plex_library`** (`makePlexLibraryComponent`, регистрируется через `Lampa.Component.add`) — тонкая обёртка над штатным `Lampa.Maker.make('Category', object)` (тот же класс, что рендерит нативные каталоги TMDB) с собственными `onCreate`/`onNext`/`onController`/`onInstance`. Карточки нормализуются в `toCard()` в минимальный набор полей, подтверждённый рабочим по коду LampaTrakt (`component`, `source`, `method`, `id`, `title`, `poster`, `image`, `release_date`, `vote_average`); `method`/`plex_type`/`plex_section` определяются **по каждому элементу** (`item.type`, `item.librarySectionID`), а не по объекту компонента — необходимо для объединённого списка, где в одной сетке вперемешку и фильмы, и сериалы из разных медиатек. Принимает `object.plex_sections` (массив ключей), `object.plex_query`, `object.plex_sort` (`{field, order}`), `object.plex_year`, `object.plex_genre_title`+`plex_genre_by_section`, `object.plex_country_title`+`plex_country_by_section`. Две стратегии загрузки (`forceClientSide` в `makePlexLibraryComponent`): **быстрый путь** — одна медиатека, без года/жанра/страны, сортировка не «Случайно»: честная постраничная загрузка и сортировка средствами самого Plex (`Api.list` с `sort=<PLEX_NATIVE_SORT_KEY[field]>:order`, быстро, не тянет всё в память); **клиентский путь** (несколько медиатек сразу, ИЛИ активен фильтр года, ИЛИ сортировка «Случайно») — Plex не даёт единого endpoint на объединённый список медиатек и не поддерживает диапазоны года как параметр запроса, поэтому `fetchCombinedItems()` собирает **все** страницы **всех** нужных медиатек (жанр/страна передаются как родные Plex query-параметры `genre=`/`country=` уже на этапе сбора — сужает объём), затем год фильтруется (`plexYearMatches`) и сортирует (`sortItems`/`plexSortValue`, включая честный Фишер-Йетс `shuffleItems` для «Случайно» — НЕ через компаратор `Math.random()` в `sort()`, это было бы некорректно) уже на клиенте, а `onNext` отдаёt уже готовые срезы из этого массива в памяти.
- **Компонент `plex_hub`** (`Lampa.Component.add`) — единственная точка входа из левого меню («Медиатека Plex» → `openLibrary()` → `Lampa.Activity.push({component:'plex_hub', plex_sections_available: chosen})`, без промежуточного выбора медиатеки). Обёртка-хаб над `plex_library` по образцу `watchlistHub` («Хочу посмотреть») из LampaTrakt: строка фильтров сверху (`controls`/`filtersRow`, `simple-button simple-button--filter selector`, подсветка активной кнопки — золотой `rgba(229,160,13,…)`/`#e5a00d` вместо фиолетового Trakt `#9f42c6`, тот же приём через `.css()` в `updateBtn`, что и в `watchlistHub`) + тело (`body`) с вложенным Category-компонентом. Ровно те же 5 кнопок в том же порядке, что и в «Хочу посмотреть», только первая — медиатека вместо типа контента:
  - **«Медиатека»** — по умолчанию «Все» (объединённый список всех выбранных медиатек). `Lampa.Select.show` со списком: **«Поиск»** первым пунктом (над «Все» — по прямой просьбе пользователя), затем **«Все»**, затем каждая медиатека отдельно — сужает список до одной. «Поиск» ищет **в текущем масштабе** (по всем медиатекам, если ничего не сужено; в рамках одной, если сужено).
  - **«Год»** — `buildYearFilterItems()`, портировано из LampaTrakt почти без изменений (те же группы: отдельные годы с 2020, затем блоки по 5 лет, затем фиксированные диапазоны до «до 1970»), без `Lampa.Lang` (в этом плагине нет слоя переводов). Фильтрует по `item.year`, попадание в диапазон — `plexYearMatches()`.
  - **«Жанр»** / **«Страна»** — реальные значения из самой медиатеки через нативные Plex-эндпоинты `Api.genres(sectionKey)`/`Api.countries(sectionKey)` (`/library/sections/{key}/genre|country`, те же, что использует официальный клиент для этих же чипов), а не захардкоженный словарь как жанры Trakt у LampaTrakt — потому что у Plex это открытый, зависящий от медиатеки список. У каждой медиатеки свои числовые ключи для одного и того же жанра/страны, поэтому `loadFilterOptions()` в хабе объединяет их **по названию** (`bySection: {title: {sectionKey: key}}`) один раз при первом открытии фильтра и кэширует на время жизни экрана; при реальном запросе каждая медиатека резолвит свой собственный ключ (`sectionExtraParams()` в `plex_library`), а медиатека без такого жанра/страны в объединённом просмотре просто пропускается (гарантированно даст 0 совпадений).
  - **«Сортировка»** — По дате добавления / По названию / По дате выхода / По длительности / Случайно / По рейтингу / Моя оценка / Просмотрено; повторный выбор того же пункта переключает направление (`switchSort` из LampaTrakt). Это ровно набор `WATCHLIST_SORT_FIELDS` из LampaTrakt **за вычетом** `WATCHLIST_VIP_SORT_FIELDS` (imdb/tmdb/RT/Metascore/голоса — Trakt VIP, у Plex таких данных и нет) и ещё трёх полей, для которых в Plex нет источника данных: `rank` («По позиции» — ручной порядок в watchlist Trakt), `popularity` (метрика самого Trakt) и `collected` (дата добавления в коллекцию Trakt — для Plex неотличимо от «Дата добавления»). Оставшиеся смэплены на реальные поля Plex: `added→addedAt`, `title→titleSort`, `released→originallyAvailableAt`/`year`, `runtime→duration`, `percentage→rating`/`audienceRating` (у Trakt это тоже общий рейтинг в процентах, а не прогресс просмотра — несмотря на название), `my_rating→userRating`, `watched→lastViewedAt`.
  Навигация вверх-вниз между строкой фильтров и сеткой — тем же приёмом, что и в LampaTrakt: свой `Lampa.Controller.add('plex_hub_controls', {...})` плюс `onController` в `plex_library`, переопределяющий `controller.up` на `object.onHead()` (переданный из хаба колбэк `Controller.toggle('plex_hub_controls')`), когда `Navigator.canmove('up')` возвращает `false` внутри сетки. Работает потому, что `items/category/base.js` эмиттит событие `'controller'` **синхронно** прямо перед `Controller.add('content', controller)` — слушатель, добавленный через `.use({onController})`, успевает подменить `controller.up` до регистрации (проверено по исходникам `yumata/lampa-source`, тот же приём, что использует `baseComponent`/`onController` в LampaTrakt).
- **Hybrid-роутинг карточки**: если у элемента Plex есть `Guid` с `tmdb://` (или legacy `guid` вида `themoviedb://...`) — карточка помечается `component:'full', source:'tmdb'` и `onEnter` делает `Lampa.Activity.push(element)`, открывая родную карточку Lampa. Иначе `onEnter` вызывает `openPlexDetailModal` — свою модалку (постер/описание/год из Plex + кнопка Play; для сериала — сезон/серия через `Lampa.Select.show`).
- **Индекс медиатеки** (`_plexTmdbIndex`, `rebuildTmdbIndex()`): персистентная карта `method:tmdbId → {ratingKey}`, обходит постранично (`fetchAllSectionItems`, размер страницы 200) все выбранные секции через `Api.list()`. Загружается из `Storage` при старте плагина, обновляется: (1) тихо при **каждом** запуске Lampa (`autoRebuildTmdbIndexOnStart`, отложен на 3с через `setTimeout`, чтобы не мешать старту приложения — без проверки на устаревание, безусловно), (2) сразу после изменения списка медиатек в `pickSections()` (только если набор реально отличается от того, что было при открытии экрана — см. `arraysEqualUnordered`), (3) вручную кнопкой «Обновить кэш медиатеки» в настройках (со статусом «когда обновлялось / сколько сопоставлено»). `toCard()` при построении сетки `plex_library` также дополняет этот же объект в памяти — обзор через «Медиатека Plex» мгновенно виден и до плановой пересборки.
- **Кнопка «Смотреть в Plex» на родной карточке**: `Lampa.Listener.follow('full', ...)` на событии `'complite'` смотрит в `_plexTmdbIndex[method+':'+id]` (не зависит от того, как пользователь попал на карточку — поиск, рекомендации, наша сетка — лишь бы индекс был собран хоть раз) и, если совпадение есть, вставляет кнопку в `.buttons--container` — тот самый контейнер, откуда меню «Смотреть» строит список источников (Торренты/Онлайн/…), обычным `.append()`. Класс кнопки `full-start__button selector` (+ маркерный `plex-watch-btn`), внутри — инлайновый `PLEX_PLAY_ICON` (отдельная константа, не `PLEX_ICON` из меню/настроек — см. запись в истории версий про новый значок) и `<span>` с подписью, обработчик — `.on('hover:enter', ...)`. Паттерн проверен и скопирован напрямую из `LampaTrakt` — функция `addAtButton(card, method)` создаёт кнопку «Авто-торрент» с классом `full-start__button selector trakt-magic-button`, а `onFullCardReady` вставляет её через `magicRoot.find('.buttons--container').append(atBtn)` (комментарий в их коде: «вставляем в `.buttons--container`, откуда «Смотреть» берёт список источников»). Более старая версия нашей кнопки вставлялась отдельным чипом после `.full-start-new__rate-line` и зависела от per-session `_plexTmdbMatchCache`, заполнявшегося только при просмотре собственной сетки плагина — оба ограничения устранены (см. запись в истории версий ниже).
- **Автозакрепление через штатный механизм Lampa** (`pinPlexAsDefaultSource`/`restorePlexPriorityIfNeeded`, оба в `initFullCardHook`): изучен `src/components/full/start/buttons.js` из `yumata/lampa-source` — родной «пин» источника хранится ГЛОБАЛЬНО в `Lampa.Storage` под ключом `full_btn_priority`, значение — `Lampa.Utils.hash(...)` от `outerHTML` конкретной кнопки (без класса `focus`); `onGroupButtons()` при каждом `emit('groupButtons')` ищет среди `.buttons--container > .full-start__button` кнопку с совпадающим хэшем и, если находит, клонирует её через `emit('priorityButton', btn)` в `.full-start-new__buttons` как `.button--priority` (это и есть визуально «закреплённая на карточке» кнопка). Критичная деталь тайминга — в `components/full.js`: `Lampa.Listener.send('full', {type:'complite', ...})` вызывается, и **на следующей же строке**, синхронно, `this.items[0].emit('groupButtons')` — то есть если внутри нашего обработчика `'complite'` мы синхронно вставили кнопку в DOM и выставили `full_btn_priority` в её хэш, `onGroupButtons()` гарантированно найдёт и закрепит именно её, без необходимости лезть во внутренний `Emit` инстанса `Start` (недоступен снаружи). `plexButtonHash()` — хэш строит один раз от статичной разметки кнопки и кэширует (разметка одинакова для любой карточки → хэш одинаков). Сохранение/восстановление предыдущего состояния — отдельный ключ `plex_prev_btn_priority`: `pinPlexAsDefaultSource()` сохраняет туда текущее значение `full_btn_priority` ТОЛЬКО если оно ещё не равно нашему хэшу (не затирает исходное предпочтение при переходе между несколькими Plex-карточками подряд), затем выставляет наш хэш; `restorePlexPriorityIfNeeded()` (вызывается в ветке «нет совпадения» в `initFullCardHook`, а также в `resetSettings()`) возвращает сохранённое значение обратно, но только если `full_btn_priority` всё ещё равен нашему хэшу (если пользователь сам вручную закрепил что-то другое уже после нас — не перезатираем его недавний осознанный выбор).
- **Воспроизведение** — только Direct Play: `Api.metadata(ratingKey)` → `Media[0].Part[0].key` → прямая ссылка с токеном в query. Субтитры — только sidecar-дорожки (`Stream.streamType === 3`), embedded/PGS не извлекаются. `Lampa.Timeline.update`+`Lampa.Timeline.view(hash)` (`hash = 'plex_' + ratingKey`) перед `Lampa.Player.play(...)` — резюме позиции. Передача во внешний плеер — целиком на стороне самой Lampa (`Storage.field('player'...)`), плагин не реализует ничего специально для этого.
- **Синхронизация прогресса**: `Lampa.Player.listener.follow('destroy', ...)` читает `Lampa.Timeline.view(hash)` и шлёт `GET /:/timeline` (+ `/:/scrobble` при ≥90%) на сервер Plex.
- **Интеграция с LampaTrakt** (опционально, настройка `plex_trakt_status_enabled`, видна только если LampaTrakt установлен):
  - Детект присутствия: `traktAvailable()` = `Lampa.SettingsApi.getComponent('trakt')` — LampaTrakt регистрирует свой settings-компонент под именем `'trakt'`. Специально НЕ используется `Lampa.Manifest.plugins` (как это делает сам LampaTrakt для детекта плагина «Applecation») — проверено, что сам LampaTrakt нигде не пишет `Lampa.Manifest.plugins = {...}` для саморегистрации, поэтому этот путь для него не сработал бы.
  - Учётные данные читаются напрямую из общего `Lampa.Storage` (тот же движок хранения, что и у LampaTrakt, всё в одном Lampa-приложении): `trakt_client_id` + разрешение активного токена в `getTraktActiveToken()`, повторяющее порядок `multiAccountGetActiveSlot`/`multiAccountGetSlot` из LampaTrakt (`trakt_accounts[].slot === trakt_active_slot`, откат на плоский `trakt_token` для немигрированных установок).
  - `traktRequest()` — прямые вызовы `api.trakt.tv` (заголовки `trakt-api-key`/`Authorization: Bearer`) в обход самого LampaTrakt, т.к. он не предоставляет публичного JS API для чтения своего кэша просмотренного.
  - `getTraktWatchedIndex()` — кэш в памяти (10 минут, `TRAKT_WATCHED_CACHE_MS`) карты `movie:tmdbId`/`tv:tmdbId` из `/sync/watched/movies` + `/sync/watched/shows`. Для сериалов «completed» считается по `plays >= aired_episodes` (та же устойчивая эвристика, что LampaTrakt применяет после того, как Trakt периодически убирает поэпизодную разбивку `seasons` из ответа — см. их же changelog v3.2.36–38), а не по наличию точной разбивки.
  - Отображение — только в собственном интерфейсе плагина (карточка `openPlexDetailModal` через `statusLineForMeta`, список серий `openEpisodePicker`), не на родной карточке Lampa — там статус уже показывает сам LampaTrakt, если установлен и авторизован.
  - `syncTraktStatusesToPlex()` — ручная (кнопка с подтверждением через `Lampa.Select.show`) односторонняя запись Trakt → Plex: фильмы и **полностью** просмотренные сериалы (все серии) из истории Trakt отмечаются просмотренными в Plex через `GET /:/scrobble`. Сопоставление — через уже собранный `_plexTmdbIndex`. Последовательно, с задержкой 150мс между запросами (не заваливать Plex-сервер). Частично просмотренные сериалы **намеренно пропускаются** — риск ошибки из-за нестабильной поэпизодной разбивки Trakt сочтён неприемлемым для необратимой записи в Plex.

## Известные ограничения (см. также README)

- Только Direct Play — нет серверного транскодирования Plex (HLS-сессии, `/video/:/transcode/universal/*`). Файлы в несовместимых кодеках (HEVC/DTS и т.п.) не воспроизводятся, если ни встроенный, ни внешний плеер их не тянут напрямую.
- Субтитры — только внешние (sidecar), не встроенные в контейнер.
- Синхронизация прогресса при внешнем плеере зависит от того, возвращает ли конкретный внешний плеер позицию обратно в `Lampa.Timeline` — как и для любых других источников в Lampa, это вне контроля плагина.
- Одна выбранная секция на попытку сохранения нескольких Plex-серверов на один аккаунт не тестировалась специально; при входе через plex.tv можно выбрать сервер из списка, переключение — через повторный вход или ручную правку адреса.

**Статусы Trakt.TV + синхронизация в Plex** — по запросу пользователя добавлена опциональная интеграция с LampaTrakt (см. раздел «Архитектура плагина» выше за техническими деталями). Настройка `plex_trakt_status_enabled` (Да/Нет, видна только если `Lampa.SettingsApi.getComponent('trakt')` найден) переключает источник статусов «просмотрено»/прогресс в собственном интерфейсе плагина (карточка, список серий) с Plex на активный аккаунт Trakt. Вторая часть просьбы — пользователь подтвердил явно (на прямой уточняющий вопрос), что нужна ещё и обратная запись: кнопка «Синхронизировать статусы в Plex» (видна только при включённом тумблере) с подтверждением через `Lampa.Select.show` перед запуском, разово отмечает в Plex как просмотренные фильмы и полностью просмотренные сериалы из истории Trakt. Это единственное место в плагине, которое пишет статус «просмотрено» в Plex массово и необратимо (не за один плейбек, как обычная синхронизация прогресса) — поэтому: только ручной запуск, только по явному подтверждению, никогда не автоматически/в фоне.

**Фикс: включил «Статусы Trakt.TV» — статусы Plex пропали, статусы Trakt не появились** — пользователь сообщил ровно это. Причина найдена в `getTraktWatchedIndex()`: оба запроса к Trakt (`/sync/watched/movies`, `/sync/watched/shows`) внутри `Promise.all` были обёрнуты каждый в свой `.catch(() => [])`. При реальном сбое (протухший токен LampaTrakt, неверный `client_id`, сетевая ошибка и т.п.) это означало, что `Promise.all` НИКОГДА не отклонялся — он благополучно резолвился пустыми массивами, строился пустой (но полноценный, truthy) индекс, и этот пустой индекс кэшировался на 10 минут как «настоящие данные Trakt». В результате внешний `.catch()` в `statusLineForMeta`/`openEpisodePicker` (задуманный как откат на Plex при сбое) никогда не срабатывал — Trakt-ветка молча использовалась с пустыми данными везде, поэтому статусы Trakt не показывались (индекс пуст), а откат на Plex не происходил (ошибки формально не было). Убраны внутренние `.catch()` у обоих `traktRequest(...)` — теперь реальный сбой корректно долетает до внешнего `.catch()` и включает откат на Plex, как и задумывалось изначально.

Заодно добавлена диагностика — пункт настроек **«Проверить подключение к Trakt»** (виден только при включённом тумблере): по нажатию форсирует запрос к Trakt и показывает результат (найдены ли `client_id`/токен, успех с числом фильмов/сериалов, либо текст ошибки — `describeAjaxError()` достаёт HTTP-статус из jqXHR). `_traktLastFetch` хранит результат последней попытки, чтобы не гадать вслепую при повторных жалобах на статусы.

**Фикс №2: пагинация Trakt + ненадёжная per-episode разбивка** — после первого фикса пользователь сообщил, что статусов всё равно нет, а диагностика показала «ок, фильмов: 100, сериалов: 100». Ровно 100/100 — сигнатура пагинации: Trakt теперь режет `/sync/watched/movies` и `/sync/watched/shows` по 100 элементов на страницу (LampaTrakt столкнулся с тем же, их changelog v3.2.39), а плагин читал только первую страницу — всё, что за пределами первой сотни, читалось как «не просмотрено» (скорее всего именно поэтому у пользователя проверяемые тайтлы были пустыми). Добавлен `traktGetAllPages(path, params)` — постраничная дочитка (`page=N&limit=100`) до короткой страницы или предохранителя в 50 страниц; используется для обоих watched-эндпоинтов. Второй пласт: `/sync/watched/shows` с `extended=full` **не отдаёт** разбивку по сериям (Trakt убрал `seasons`, LampaTrakt changelog v3.2.38), поэтому старый per-episode статус в списке серий был всегда пуст. Теперь точный per-episode статус берётся из отдельного эндпоинта `/shows/{traktId}/progress/watched` (`getTraktShowEpisodeSet` — резолвит `tmdbId → traktId` из watched-индекса, иначе через `/search/tmdb/{id}?type=show`, затем читает `seasons[].episodes[].completed`; кэш в памяти на 10 мин). В watched-индексе теперь хранится `traktId` каждого сериала (для этого резолва), а бесполезная попытка собрать `episodes` из watched убрана. **Важно:** Trakt отслеживает серии бинарно (просмотрено/нет), без mid-episode percent — поэтому при включённых статусах Trakt в списке серий вместо процента Plex показывается метка «Просмотрено (Trakt)» либо пусто; resume-процент — это концепт Plex/Timeline, у Trakt его нет.

**Мелкие правки по обратной связи (статусы работают)** — пользователь подтвердил, что после фикса №2 статусы Trakt показываются корректно (диагностика: фильмов 2688, сериалов 342). Два уточнения:
1. Убран суффикс «(Trakt)» из всех строк статуса (`statusLineForMeta`, список серий) — лишний шум, источник и так понятен из настройки.
2. **Apple TV: кнопка «Смотреть в Plex» не подсвечивалась навигацией.** Причина — оставшийся с ранней версии `setTimeout(() => Lampa.Controller.collectionSet(root), 0)` в `injectPlexButton`. После перехода на штатное закрепление (`full_btn_priority`) навигацию/фокус карточки полностью настраивает сама Lampa в `Start.toggle` (`collectionSet` + `collectionFocus`), а наш отложенный `collectionSet` выполнялся ПОЗЖЕ и пересобирал коллекцию БЕЗ восстановления фокуса — на десктопе/тач незаметно, а на Apple TV пропадала подсветка выделения. Вызов убран; кнопку просто добавляем в скрытый `.buttons--container`, остальное делает Lampa. Заодно поправлено мёртвое CSS-правило иконки кнопки (ссылалось на класс `.plex-brand-icon`, которого нет у нового `PLEX_PLAY_ICON`) → теперь `.full-start__button.plex-watch-btn svg`.

**Переименование кнопки + позиция воспроизведения (resume)** — два запроса пользователя:
1. Кнопка «Смотреть в **в** Plex» → «Смотреть **из** Plex» (Plex в Lampa — это источник воспроизведения, а не место, куда переходят). Переименованы обе точки использования метки (реальная кнопка `injectPlexButton` + «пробный» элемент `plexButtonHash`) — они обязаны совпадать байт-в-байт, иначе штатное закрепление Lampa (`full_btn_priority`) перестанет находить кнопку.
2. **Resume не работал** — фильм/серия с частичным прогрессом (напр. 54%) начинались с начала, и во встроенном, и во внешнем плеере. Изучено, как это сделано в LampaTrakt: он **не вызывает `Player.play` напрямую** — полагается на нативное воспроизведение торрентов Lampa и лишь записывает позицию в хранилище `file_view_<profile>` по хэшу файла (`setFileViewProgress`), чтобы нативный плеер сам подхватил её. Наш случай иной (мы зовём `Player.play` со своим URL и объектом `timeline`), поэтому для внутреннего плеера resume должен работать через `data.timeline.time`. Вероятная причина сбоя — повторный запрос `Api.metadata` иногда возвращал элемент без `viewOffset`. Фикс: `playMeta(meta, cardData, playlistMetas, resume)` — новый параметр `resume {viewOffset, duration}` (в мс) с приоритетом над `meta.viewOffset`; для серий он берётся напрямую из элемента списка (`episodeStub`), по которому пользователь и видел процент — позиция гарантированно совпадает с показанной. Внутренний плеер перематывает по `timeline.time`, Android/webOS-внешние плееры получают позицию из `data.timeline.time` (Lampa проставляет `data.position` из неё). **Ограничение Apple TV:** для внешних плееров Apple TV (infuse/tvos и др.) сама Lampa НЕ передаёт стартовую позицию в URL-схеме (`infuse://…play?url=…`, `lampa://video?player=tvos&src=…` — параметра позиции там нет), в отличие от Android/webOS; resume в них зависит от собственной памяти плеера (infuse помнит позицию по URL). Это ограничение платформы/Lampa, не плагина.

**Фикс resume для внешних плееров Apple TV (tvOS Pro/infuse) — предыдущий вывод об «ограничении Apple TV» был НЕВЕРНЫМ.** Пользователь резонно заметил: при воспроизведении торрентов на tvOS Pro resume работает (стартует с середины). Значит позиция ДО плеера доходит, просто не через тот механизм, что я думал. Разобрано в `src/interaction/torrent.js` (нативное воспроизведение торрентов): воспроизводимый `element` содержит `element.playlist`, куда пушится в т.ч. **сам текущий элемент**, и **каждый элемент playlist несёт собственный `timeline`** (`{title, url, timeline, season, episode, ...}`). Внешний плеер (tvOS Pro/infuse) сопоставляет проигрываемый `src`-url с элементом playlist по url и берёт стартовую позицию из его `timeline` — позиция едет внутри `playlist` (сериализуется в URL-схему как `&playlist=`), а НЕ в отдельном параметре позиции. Наш баг: мы клали в `playData.playlist` только СЛЕДУЮЩИЕ серии, без текущего элемента и без `timeline` у элементов — плееру нечего было сопоставить. Фикс: `buildPlexPlaylistItem(meta, resume)` собирает элемент `{title, url, timeline, subtitles}` (timeline через `Lampa.Timeline.update`+`view` по позиции из resume/meta), а `playMeta` формирует `playlist`, начиная с **текущего** элемента (его `url === playData.url`), затем следующие серии — у каждой свой `timeline`. Теперь позиция доезжает до внешних плееров Apple TV так же, как у торрентов. Также это чинит корректность навигации по плейлисту (раньше текущего элемента в playlist не было вовсе).

**Отправка просмотра из Plex в Trakt** — по запросу пользователя. Решён архитектурный вопрос «где делать»: на стороне **Lampa-Plex**, т.к. только наш плагин знает в момент воспроизведения, что именно играет (tmdb id + сезон/серия); LampaTrakt про Plex-воспроизведение не знает (его скробблинг привязан к торрент-файлам через кэш хэшей), и учить его этому — лишняя связанность + правки чужого репозитория. Токен обновляет сам LampaTrakt (пишет свежий в `Storage`), мы читаем текущий. Реализация: тумблер `plex_trakt_scrobble` («Отправлять просмотр в Trakt», виден при установленном LampaTrakt). В `initTimelineSync` на `Player 'destroy'` для каждого реально просмотренного элемента (`road.time>0`) вызывается `reportPlaybackToTrakt`: при ≥90% — `POST /sync/history` (отметка «просмотрено»), ниже — `POST /scrobble/pause` (позиция для resume, best-effort). `traktPost` — POST-версия `traktRequest`. Идентификатор для Trakt (`{type:'movie',tmdb}` / `{type:'episode',tmdb:<show>,season,number}`) прокидывается через `playMeta(...,trakt)`: для фильмов из `injectPlexButton` (`e.data.id`), для серий из `showMeta`+`episodeStub` (`parentIndex`/`index`). **Очередь повтора** (`plex_trakt_scrobble_queue` в `Storage`): при сбое `POST /sync/history` payload сохраняется и до-отправляется на старте (`flushTraktQueue`, отложен 5с) — на прямой вопрос пользователя «при каком следующем разе, пересматривать?» ответ: НЕТ, повтор автоматический, пересмотр не нужен. После успешной отметки — `invalidateTraktCaches()` (сброс `_traktWatchedIndex`/`_traktEpisodeSetCache`), чтобы интерфейс плагина сразу показал новый статус. **Бинж-просмотр**: `_activePlexPlayback.items` хранит `{ratingKey,trakt}` по всем элементам плейлиста (текущий + следующие серии), поэтому при автопереходе отмечаются все просмотренные серии, а не только первая (актуально для внутреннего плеера; внешние обычно не отдают попутный прогресс назад).

Несколько сразу — Plex не даёт объединённого listing-endpoint, поэтому `fetchCombinedItems()` (новый, на основе обобщённого `fetchAllSectionItems(sectionKey, query, onPage)` — раньше без параметра `query`, использовался только для сборки tmdb-индекса) собирает все страницы всех выбранных медиатек и сортирует/пагинирует уже на клиенте (`sortItems()`/`plexSortValue()`). `toCard()` перестал зависеть от `object.plex_type`/`object.plex_section` — теперь определяет `method`/`plex_type`/`plex_section` **по каждому элементу** (`item.type`, `item.librarySectionID`), что обязательно для объединённой сетки с разнородным контентом.

**Полная параллель с «Хочу посмотреть» LampaTrakt: 5 кнопок вместо 2, жанр/страна, весь набор не-VIP сортировок** — пользователь уточнил предыдущую реализацию `plex_hub`: нужны были ВСЕ те же кнопки, что и в `watchlistHub`, в том же порядке (медиатека вместо типа контента → год → жанр → страна → сортировка), тот же визуальный приём подсветки активной кнопки (просто в цвете Plex вместо Trakt), и тот же набор видов сортировки за вычетом Trakt VIP.
- **Год**: `buildYearFilterItems()` + `yearFilterLabel()` портированы из LampaTrakt дословно (без `Lampa.Lang` — в этом плагине i18n-слоя нет). Фильтр (не сортировка) — сужает уже собранный на клиенте список по `item.year` (`plexYearMatches()`), одиночный год или диапазон вида `"2020-2024"`.
- **Жанр/Страна**: у Plex, в отличие от Trakt, это открытый список, зависящий от конкретной медиатеки, а не фиксированный словарь — поэтому вместо захардкоженного `TRAKT_GENRE_NAMES_RU`-аналога добавлены `Api.genres(sectionKey)`/`Api.countries(sectionKey)` (`GET /library/sections/{key}/genre|country` — те же нативные Plex-эндпоинты «списка значений фильтра», что использует официальный клиент). У каждой медиатеки свои числовые ключи для одного и того же жанра, поэтому `loadFilterOptions()` в `plex_hub` объединяет их по **названию** (`{titles, bySection: {title: {sectionKey: key}}}`), загружается один раз при первом открытии соответствующего фильтра и кэшируется на время жизни экрана. При построении запроса каждая медиатека резолвит СВОЙ ключ (`sectionExtraParams()` внутри `makePlexLibraryComponent`); медиатека, в которой такого жанра/страны нет вовсе, в объединённом просмотре пропускается целиком (гарантированно дала бы 0 совпадений) — не шлём заведомо бессмысленный запрос.
- **Стратегия загрузки пересмотрена**: раньше клиентский (fetch-all + сортировка в памяти) путь включался только при нескольких медиатеках сразу. Теперь `forceClientSide` в `makePlexLibraryComponent` дополнительно учитывает активный фильтр года (Plex не поддерживает диапазоны в query-параметре `year`, только точное совпадение — рисковано полагаться на недокументированное поведение) и сортировку «Случайно» (нет гарантии, что `sort=random` поддерживается всеми версиями Plex Media Server). Жанр/страна, наоборот, спокойно передаются как родные Plex-параметры (`genre=`/`country=`) уже на этапе постраничного сбора данных (`Api.list` → `fetchAllSectionItems` → и туда, и в «быстрый путь» одной медиатеки) — сужают объём без похода в клиентскую фильтрацию.
- **Сортировка «Случайно»**: реализована явным перемешиванием (`shuffleItems()`, Фишер-Йетс), НЕ через `Math.random()` внутри компаратора `Array.sort` — последнее вызывало бы `plexSortValue` по несколько раз на один и тот же элемент с разным результатом при каждом сравнении, что ломает транзитивность отношения порядка и даёт по сути неопределённый результат сортировки. Повторный выбор «Случайно» в меню сортировки естественным образом даёт новую перетасовку — `rebuildView()` создаёt новый экземпляр `plex_library` с чистым `mergedItems`.
- **Набор сортировок** — `PLEX_SORT_ORDER`/`PLEX_SORT_LABELS`: `added` (По дате добавления), `title` (По названию), `released` (По дате выхода), `runtime` (По длительности), `random` (Случайно), `percentage` (По рейтингу — у Trakt это тоже общий процентный рейтинг, а не прогресс просмотра, несмотря на название), `my_rating` (Моя оценка), `watched` (Просмотрено/дата последнего просмотра). Это ровно `WATCHLIST_SORT_FIELDS` из LampaTrakt **минус** `WATCHLIST_VIP_SORT_FIELDS` (imdb/tmdb/RT/Metascore/голоса разных источников — Trakt VIP, у Plex таких данных нет в принципе) **минус** три поля без источника данных в Plex: `rank` («По позиции» — ручной порядок в watchlist Trakt, у Plex нет аналога), `popularity` (внутренняя метрика популярности самого Trakt) и `collected` (дата добавления в коллекцию Trakt — для Plex неотличимо от «Дата добавления», отдельного понятия «коллекция» с собственной датой у Plex нет). Для «быстрого пути» (одна медиатека, никаких фильтров, не «Случайно») используется нативная Plex-сортировка через `PLEX_NATIVE_SORT_KEY` (`added→addedAt`, `title→titleSort`, `released→originallyAvailableAt`, `runtime→duration`, `percentage→rating`, `my_rating→userRating`, `watched→lastViewedAt`) — сервер и сортирует, и постранично отдаёт результат сам, без «съедания» всей медиатеки в память.
- Дизайн кнопок — `simple-button simple-button--filter selector` + класс `plex-hub__filter`; подсветка активной кнопки в золотом Plex (`rgba(229,160,13,.18)` фон / `#e5a00d` нижняя рамка) вместо фиолетового Trakt. Разобрано по большому `<style>`-блоку (`Lampa.Template.add('trakt_style', …)`) в самом LampaTrakt:
  - **Одна строка вместо переноса**: у LampaTrakt кнопки — `flex:1 1 0; min-width:0` (делят строку поровну и ужимаются, НИКОГДА не переносятся; длинный текст переносится ВНУТРИ кнопки через `white-space:normal; word-break:break-word`). Я ошибочно ставил `flex:1 1 auto` (basis = ширина контента) + `flex-wrap:wrap` — из-за чего 5 кнопок не влезали и заворачивались на второй ряд.
  - **Нет «коробки» у неактивных**: нативный Lampa `.simple-button--filter` рисует свою «коробку» через фон у ВНУТРЕННЕГО `> div`. LampaTrakt глушит её правилом `.simple-button--filter>div{background:transparent!important}` — остаётся только текст. Я этого правила не воспроизвёл, поэтому нативная коробка (в теме этого устройства — золотистая) показывалась у всех кнопок. Добавил аналогичное правило для `.plex-hub__filter .plex-hub__filter-label, .plex-hub__filter.simple-button--filter>div{background:transparent!important}`.
  - Стили полностью перенесены из инлайн-`.css()` в CSS-блок `injectStyles()` (мэппинг классов `.trakt-watchlist-hub__sorts`→`.plex-hub__filters`, `.trakt-watchlist__sort`→`.plex-hub__filter`, `.trakt-watchlist__sort--active`→`.plex-hub__filter--active`, `.focus/.hover` — как у LampaTrakt), `updateBtn` теперь только тогглит класс `--active` (без инлайна), `makeBtn` не задаёт инлайн-стилей вовсе. Цвет активной кнопки/фокуса — золото Plex вместо фиолетового Trakt, остальное 1:1.
  - Перед пушем собрано HTML-превью строки фильтров (тот же CSS) и отправлено пользователю на подтверждение — рендер реального Lampa-клиента в песочнице по-прежнему недоступен.

**Фикс высоты кнопок фильтров (по скриншоту с устройства)** — после предыдущего фикса стало похоже на LampaTrakt, но кнопки оказались слишком высокими. Две причины: (1) длинная подпись сортировки «По дате добавления ↓» переносилась в 3 строки в узкой равной колонке (`flex:1 1 0` делит строку на 5 равных долей); (2) контейнер `.plex-hub__filters` по умолчанию `align-items:stretch` растягивал ВСЕ кнопки по высоте самой высокой (той самой 3-строчной сортировки), поэтому даже «Все» (1 строка) раздувалась. Первая попытка (промежуточная): `align-items:center`, шире сортировка `flex:1.9 1 0`, меньше отступ/шрифт — помогло, но не до конца.

**Фикс: кнопки по ширине контента + `space-between` (финальный подход к строке фильтров)** — на следующем скриншоте всё ещё переносились «Стран/а» и сортировка в несколько строк, при явно свободном месте по бокам. Корень: любой `flex:1 1 0`/`flex:N 1 0` жёстко нарезает строку на доли, и КАЖДАЯ подпись обрезается до ширины своей доли — короткие («Все»/«Год»/«Жанр») зря занимают полосу, а «Страна»/сортировка не влезают и переносятся. LampaTrakt (по скриншоту с реального устройства — важно: рендер отличается от CSS в scratchpad-версии `trakttv.js`, там `flex:1 1 0`, но на девайсе кнопки явно по ширине контента) размещает кнопки **по ширине текста** и разносит по строке. Перешёл на эту модель: `.plex-hub__filter{flex:0 0 auto}` (ширина = контент, без нарезки на доли), подпись `white-space:nowrap` (никогда не переносится), контейнер `justify-content:space-between` (первая кнопка слева, сортировка справа, средние равномерно между — как в «Хочу посмотреть»). `flex-wrap:wrap` оставлен как fallback: если на совсем узком экране в строку не влезает, кнопки перенесутся целиком (а не порежут текст). Убран класс `--sort` и его правило ширины (при контентной ширине сортировка сама ровно по своему тексту). Обновлённое превью повторно отправлено пользователю. НЕ ПОМОГЛО: на устройстве сортировка всё равно ушла на вторую строку (контентные ширины 5 кнопок с отступами оказались ~97% строки — впритык не влезло), см. следующую запись.

**Финальный фикс строки фильтров: гарантированная одна строка (`nowrap` + рост как у LampaTrakt + ellipsis)** — пользователь резонно спросил, почему просто не скопировать код LampaTrakt. Разбор: код и был скопирован (инлайновый `flex:1 1 auto` из `makeBtn` LampaTrakt + `flex-wrap:wrap` контейнера — проверено по `trakttv.js:5186-5192`, инлайн переопределяет `flex:1 1 0` из их стилевого блока), но у LampaTrakt на скриншоте пользователя всё влезло в одну строку только потому, что его текущая подпись сортировки короче («По дате выхода ↓»), а наша дефолтная — «По дате добавления ↓», на 4 символа длиннее; этих пикселей и не хватало, отчего каждая итерация с «переносимой» раскладкой (`wrap`) заканчивалась второй строкой. Итог — та же модель, что у LampaTrakt (кнопки `flex:1 1 auto`: растут и заполняют строку, сортировка визуально шире остальных), но однострочность обеспечена жёстко, без зависимости от длины подписи: контейнер `flex-wrap:nowrap`, кнопки `min-width:0` (могут ужиматься), подпись `white-space:nowrap; overflow:hidden; text-overflow:ellipsis` (в худшем случае обрежется многоточием, но строка не разорвётся), шрифт `.9em` и горизонтальный отступ `.7em`, чтобы «По дате добавления ↓» помещалась целиком без ellipsis. РЕЗУЛЬТАТ: строка стала одной, но ВСЕ подписи обрезались в «В…»/«Г…»/«Жа…» — см. следующую запись, настоящий корень был не в раскладке.

**Настоящий корень проблемы строки фильтров: два родных стиля Lampa, которые LampaTrakt нейтрализует** — после скриншота с «В… Г… Жа…» скачан родной CSS Lampa (`yumata/lampa-source`, `public/css/app.css`) и извлечены все правила `.simple-button*`. Найдены два виновника всей многосерийной эпопеи:
1. `.simple-button{margin-right:1em; font-size:1.3em; height:2.8em; ...}` — родной правый отступ у КАЖДОЙ кнопки. LampaTrakt его гасит (`.trakt-watchlist__sort{margin:0;...}`), у нас он оставался: 5 кнопок × 1.3em ≈ четверть строки уходила на пустые поля (это и были «большие зазоры» на скриншотах), из-за чего контент ужимался и обрезался. Заодно родной `height:2.8em` — источник изначальной «слишком высокой» кнопки (мы уже гасили `height:auto!important`).
2. `.simple-button--filter > div{font-size:.7em; padding:.3em .5em; background:rgba(255,255,255,.3); margin-left:1em; overflow:hidden; text-overflow:ellipsis; white-space:nowrap}` — у родной подписи размер `.7em` (от кнопочных `1.3em` ≈ 0.91 базового em). LampaTrakt свой размер НЕ задаёт вовсе — наследует эти родные `.7em` (проверено: в trakttv.js нет ни одного font-size на sort-label, только на стрелку-состояние). Я же ставил `.9em`, что при базе кнопки 1.3em давало эффективные ~1.17em — текст на ~30% крупнее, чем у LampaTrakt; пользователь это в итоге и заметил визуально («шрифт сильно меньше»). Именно поэтому «та же информация» у LampaTrakt влезала, а у нас нет.
Фикс: `margin:0!important` на кнопке + `font-size:.7em` на подписи (= родной размер, как у LampaTrakt), паддинг `.5em .9em`, остальная модель без изменений (nowrap + `flex:1 1 auto` + `min-width:0` + ellipsis-страховка). Превью-файл теперь включает симуляцию родных стилей Lampa (`.simple-button`/`.simple-button--filter>div`) — прошлые превью расходились с устройством именно потому, что рендерились без них. Урок на будущее: при стилизации поверх родных классов Lampa СНАЧАЛА читать их реальный CSS (`public/css/app.css`), а не только CSS плагина-образца — плагин-образец может молча полагаться на родные значения (как тут с font-size) или нейтрализовать родные (как тут с margin), и «копия только его правил» не воспроизводит картинку.

**Значок левого меню → тот же, что у кнопки «Смотреть из Plex»; название пункта меню «Медиатека Plex» → «Plex»** — по прямой просьбе пользователя. `ICON_MENU` был `PLEX_ICON` (логотип-вихрь Simple Icons), стал `PLEX_PLAY_ICON` (play-в-квадрате — та же константа, что и в разметке кнопки на карточке); `PLEX_PLAY_ICON` для этого пришлось объявить РАНЬШЕ `ICON_MENU`/`ICON_SETTINGS` в файле (раньше он был объявлен ниже — ссылка на ещё не проинициализированную `var` дала бы `undefined` в значке меню). `ICON_SETTINGS` не тронут — логотип в настройках остаётся прежним, менялся только пункт левого меню. `Lampa.Menu.addButton(ICON_MENU, 'Медиатека Plex', ...)` → `'Plex'`, заодно и заголовок самой открываемой страницы (`Lampa.Activity.push({component:'plex_hub', title:...})`) переименован туда же для консистентности (иначе пункт меню назывался бы «Plex», а страница внутри — по-прежнему «Медиатека Plex»). README обновлён везде, где упоминался пункт меню по имени; внутренние идентификаторы (`plex_hub`, `openLibrary`, комментарии-заголовки разделов кода) не трогал — переименование чисто в пользовательских строках.

**Фикс: значок в левом меню на фокусе превращался в «тёмный квадрат без стрелки»** — пользователь сообщил сразу после предыдущей записи. Причина: `PLEX_PLAY_ICON` (использованный для `ICON_MENU`) состоит из ДВУХ элементов — `<path fill="currentColor">` (стрелка, корректно перекрашивается) и `<rect stroke="#fff">` (декоративная рамка вокруг, дизайн специально фиксировал белый цвет «чтобы форма всегда читалась чётко» на кнопке карточки — см. более раннюю запись про этот значок). `currentColor` — единственный канал, которым Lampa перекрашивает содержимое пункта меню под контраст с подсветкой на фокусе (тот же принцип, что и у рабочего `PLEX_ICON`/`plex-brand-icon`, и у `trakt-brand-icon` в LampaTrakt — оба ЦЕЛИКОМ `currentColor`, без единого фиксированного цвета). Фиксированная белая рамка в этот механизм не входит и на фокусе, видимо, накладывается на подсветку так, что стрелка перестаёт читаться. Фикс — новая константа `PLEX_MENU_ICON`: та же стрелка, тот же класс `plex-brand-icon`, что и у `PLEX_ICON`, но БЕЗ рамки — целиком `currentColor`, как и остальные рабочие значки. `ICON_MENU = PLEX_MENU_ICON`. `PLEX_PLAY_ICON` (с рамкой) не тронут — он остаётся у кнопки «Смотреть из Plex» на карточке, где фон кнопки/подсветка устроены иначе и рамка уже подтверждена пользователем как работающая корректно.

**Индикатор загрузки при открытии сезонов/серий** — пользователь сообщил, что после «Смотреть из Plex» → выбор сезона список серий открывается заметно небыстро (в этой точке последовательно: `Api.children(season.ratingKey)`, затем при включённых статусах Trakt ещё и `getTraktShowEpisodeSet()`) без вообще какой-либо индикации, что что-то происходит — выглядит как зависание. Попросил: спиннер по центру с небольшим затемнением фона на время загрузки, закрываемый тапом в любое место или «назад» на пульте (в т.ч. Apple TV), с прерыванием самой загрузки.
- Новый переиспользуемый хелпер `showPlexLoader()`/`hidePlexLoader()`: full-screen `.plex-loading-overlay` (`position:fixed`, `rgba(0,0,0,.4)` — «небольшое» затемнение, не глухое) со спиннером-кольцом в центре (золотая верхняя дуга `#e5a00d`, вращение через `@keyframes`, с `-webkit-`-дублем кадров и transform для старых WebKit смарт-ТВ). `showPlexLoader()` возвращает объект `{cancelled:false}` — вызывающий код после `await`/`.then()` каждого шага проверяет `loader.cancelled` и, если true, просто ничего не делает (не открывает Select, не показывает Noty) — сам HTTP-запрос при этом долетает до конца в фоне и его результат просто игнорируется: настоящий abort не пробрасывается через `$.ajax`-обёртку `plexRequest` (потребовал бы протащить `AbortController`/`jqXHR` через все слои `Api`/`traktRequest`), а UI и так закрывается мгновенно — ощущается пользователем как прерывание, чего и просили.
- Закрытие: (1) обычный `click` на самом оверлее (тач/мышь — работает независимо от фокуса, так как это прямой DOM-обработчик, а не завязка на `.selector`-коллекцию Lampa); (2) физическая кнопка «назад»/menu на пульте — через временный `Lampa.Controller.add('plex_loader_ctrl', {back: hidePlexLoader, ...no-op для up/down/left/right/toggle})` + `Controller.toggle('plex_loader_ctrl')` на время показа. Пульт Apple TV не имеет «тапа в произвольную точку экрана» в терминах DOM-координат (трекпад — это фокус+клик, а фокусировать в оверлее нечего), поэтому для него единственный практический путь закрытия — именно «назад»; для тач/мыши работает клик где угодно. `hidePlexLoader()` после закрытия всегда возвращает `Controller.toggle('content')` — тот же приём, что уже используется в файле при закрытии модалок/Select (`Lampa.Modal.close(); Lampa.Controller.toggle('content')`), безопасно даже перед немедленным открытием следующего `Lampa.Select.show()` (тот сразу же перехватывает контроллер на себя).

**Бейджи LampaTrakt (просмотрено/смотрю/хочу посмотреть/дата релиза) теперь появляются и на сетке «Медиатека Plex»** — задача пришла оформленным промптом от пользователя, описывающим фикс, уже сделанный на стороне самого LampaTrakt (отдельный репозиторий `KZuev/LampaTrakt`, файл `trakttv.js`, версия v3.2.50+, в этой сессии недоступен — работали строго по присланному описанию, не трогая LampaTrakt). Суть: LampaTrakt вешает свои бейджи не через общее событие Lampa `'catalog'`/`'line'` (оно ловит только карточки нативных строк главной), а напрямую внутри `onInstance` каждого собственного хаба — поэтому карточки чужих сеток (в т.ч. нашей `plex_library`) он никогда не видел, хотя его внутренние кэши просмотренного/watchlist уже заполнены корректно. Фикс на стороне LampaTrakt публикует наружу `window.TraktTV.applyBadges(cardInstance)` — тонкую обёртку над теми же render-функциями, что использует его собственный `onInstance`; безопасна для повторного вызова (не дублирует бейджи) и no-op, если LampaTrakt не установлен либо у `cardInstance.data` нет `id`/`method`/`card_type`.
В `makePlexLibraryComponent`'s `onInstance` (сразу после существующего `card.use({onEnter, onFocus})`) добавлен вызов:
```js
if (window.TraktTV && typeof window.TraktTV.applyBadges === 'function') {
    window.TraktTV.applyBadges(card);
}
```
Проверено, что предпосылки уже выполнены самим `toCard()` без доп. правок: для tmdb-сопоставленных элементов Plex туда пишутся ровно `id` (tmdb id), `method` и `card_type` (`'movie'`/`'tv'`) — те самые поля, которые ищет `applyBadges`. Для элементов без сопоставления с TMDB (нет `Guid` с `tmdb://`) этих полей у карточки нет вовсе — `applyBadges` в таком случае тихо ничего не делает, это ожидаемо (бейджи для несопоставленного Plex-контента принципиально невозможны, у LampaTrakt нет способа связать его с записью в Trakt). Интеграция полностью пассивна с нашей стороны — извлечь бейджи из этой сетки раньше или помимо `applyBadges` мы не можем, т.к. рендер-логика бейджей целиком внутри LampaTrakt.

**Диагностика и фикс регрессии бейджей LampaTrakt (после обновления LampaTrakt до v3.2.51 бейджи пропали полностью)** — задача снова пришла оформленным промптом от пользователя с историей тестов: сразу после добавления вызова показывались «Просмотрено»/«Смотрю», но не «Хочу посмотреть»; после обновления LampaTrakt до v3.2.51 (фикс на стороне LampaTrakt, недоступного в этой сессии — репозиторий `KZuev/LampaTrakt` вне scope) не показывается вообще ничего, при этом родные экраны самого LampaTrakt (Мои сериалы, Смотреть дальше, главная) бейджи показывают корректно — то есть регрессия именно в том, как/когда Lampa-Plex вызывает `applyBadges`, а не в LampaTrakt.
- Диагностировать живьём (как просил пользователь — `console.log`/`Noty` перед вызовом) в этой песочнице невозможно (нет реального Lampa-клиента), поэтому причина искалась статическим разбором родного `create.js` из `yumata/lampa-source` (уже скачивался ранее в этой сессии для другой задачи, см. записи про `plex_hub`): `onCreateAndAppend` вызывает `this.emit('instance', item, element)` (это и есть наш `onInstance`) **ДО** `item.create()` — то есть на момент нашего вызова `window.TraktTV.applyBadges(card)` DOM самой карточки (`card.render()`) ещё не построен `Card.create()`. Гипотеза: `applyBadges` не находит, куда вставлять DOM бейджей, и молча ничего не делает; в v3.2.50, судя по частичному успеху («Просмотрено»/«Смотрю» показывались), внутренняя реализация была снисходительнее к этому (может быть, ждала/ретраила), а в v3.2.51 (чинили именно watchlist-бейдж) эта снисходительность, видимо, пропала и для остальных типов бейджей тоже — отсюда полная регрессия сразу после обновления LampaTrakt, без единой нашей правки.
- Фикс: вызов `applyBadges` перенесён из СИНХРОННОЙ части `onInstance` в `setTimeout(fn, 0)` — гарантированно попадает уже ПОСЛЕ синхронного `item.create()` в том же тике события. Обёрнут в `try/catch` (третья сторона — не должны ронять наш рендер сетки, если её код когда-нибудь бросит исключение). Добавлена диагностика `_plexBadgesLastAttempt` (`{at, available, id, method, error}`), обновляемая при каждой попытке, и новый пункт настроек **«Проверить бейджи LampaTrakt на медиатеке»** (в разделе «Прочее», виден при `traktAvailable()` — НЕ гейтится тумблером `plex_trakt_status_enabled`/«Статусы Trakt.TV», т.к. бейджи работают независимо от него) — показывает: не найден ли `applyBadges` вовсе, бросил ли он исключение, или отработал без ошибок с каким `id`/`method`. Это даёт пользователю возможность самому диагностировать на устройстве без консоли/логов, если фикс не решит проблему до конца — просил именно такую возможность в постановке задачи.
- Проверено отдельно (пункт 5 из постановки, исторический — про исчезновение бейджей после смены сортировки до перезапуска приложения): начиная с редизайна `plex_hub` в этой же сессии, `rebuildView()` при любой смене фильтра/сортировки полностью уничтожает (`currentView.destroy()`) и заново создаёт (`makePlexLibraryComponent(...).create()`) весь Category-компонент — то есть карточки пересоздаются с нуля и `onInstance`/`applyBadges` естественным образом отрабатывает заново на каждую. Отдельного фикса для этого пункта не потребовалось — архитектура после редизайна это уже покрывает; если проблема исторически была актуальна, то относилась к более старой версии компонента (до `plex_hub`), которой сейчас в коде уже нет.
- Не проверялось на реальном устройстве (нет способа воспроизвести живой Lampa+LampaTrakt v3.2.51 в этой песочнице) — гипотеза о причине основана на чтении родного исходника Lampa, а не на подтверждённой трассировке. Если после этого фикса бейджи всё ещё не появляются — новый пункт настроек покажет, доходит ли вообще вызов и что происходит внутри, это и должно стать следующей зацепкой.

**Сортировка по умолчанию в настройках + локализация жанров/стран** — три отдельных запроса пользователя одним сообщением.
1. **Сортировка по умолчанию**: новый пункт `plex_default_sort` («Сортировка по умолчанию») в `Настройки → Plex → Медиатеки`, сразу после «Выбрать медиатеки». Хранится в `Storage` (`plex_default_sort_field`/`plex_default_sort_order`) через новые `getDefaultSort()`/`setDefaultSort()`; `plex_hub`'s `activeSort` теперь инициализируется через `getDefaultSort()` вместо жёстко зашитого `{field:'added', order:'desc'}` (этот хардкод остался только как fallback ВНУТРИ `getDefaultSort()`, если в `Storage` ничего не сохранено или сохранено что-то не из `PLEX_SORT_ORDER`). Выбор сортировки в настройках (`openDefaultSortPicker()`) — точная копия логики `openSortMenu()` внутри `plex_hub` (тот же список `PLEX_SORT_ORDER`/`PLEX_SORT_LABELS`, та же формула переключения направления при повторном выборе того же пункта: `order = (current.field===a.field && current.order==='asc') ? 'desc' : 'asc'`), но пишет в `Storage` вместо локального `activeSort` и обновляет статус-строку настройки через `Lampa.Settings.update()`. Общий кусок форматирования (`PLEX_SORT_LABELS[field] + ' ' + arrow`) вынесен в `sortLabelWithArrow(sort)` — переиспользуется и настройками, и `plex_hub`'s `getSortLabel()` (был почти дословный дубль, теперь одна реализация).
2. **Жанры/страны показывались на английском (или вперемешку)**: причина — Plex локализует стандартные названия из своей внутренней таблицы синонимов жанров/стран (эндпоинты `/library/sections/{key}/genre|country`) по заголовку/параметру `X-Plex-Language`, а плагин его никогда не передавал (весь остальной интерфейс уже жёстко на русском, но Plex-то об этом не знал). Добавлено `'X-Plex-Language': 'ru'` в `plexAuthParams()` — единая точка, откуда авторизационные/идентификационные параметры Plex расходятся по ВСЕМ запросам через `plexUrl()`/`plexRequest()` (как и остальные `X-Plex-*`, передаётся в query-строке, не заголовком — та же причина, что и у токена: не упираться в CORS-preflight на кастомных заголовках). Должно затронуть не только жанры/страны, но и вообще любые Plex-строки, зависящие от языка на стороне сервера (если такие есть).
3. Отдельного объяснения не нашлось для «жанры вперемешку — часть на русском, часть на английском» ДО этого фикса, кроме гипотезы, что нестандартные/редкие жанровые теги (сохранённые агентом метаданных как есть, без соответствия в таблице синонимов Plex) не переводятся в принципе ни при каком `X-Plex-Language` — это ограничение самого Plex API, а не то, что можно исправить на стороне клиента. Если после фикса останутся отдельные англоязычные жанры — это, вероятно, именно такой случай, а не баг плагина.
- Не проверялось на реальном устройстве ни по одному из трёх пунктов (нет живого Lampa+Plex-сервера в этой песочнице) — особенно неопределённость по п.2/3: `X-Plex-Language` — по опыту работы с Plex API ожидаемо влияет на локализацию таких facet-списков, но это не задокументированное публично поведение, которое можно было бы сверить без реального сервера.
